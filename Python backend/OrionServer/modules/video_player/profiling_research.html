⚠ Problem Statement

How Open Platforms Recommend Study Content

A deep dive into YouTube’s recommendation architecture — and why engagement-first algorithms fail learners.

YouTube’s Architecture: The Two-Stage DNN Pipeline

YouTube’s recommendation system, documented in Google’s landmark 2016 paper “Deep Neural Networks for YouTube Recommendations” (Covington et al.), is a two-stage deep learning pipeline designed for one primary objective: maximize watch time.

🌐 Stage 1 Candidate Generation
→
📊 Stage 2 Ranking Network
→
📺 Sidebar ~20 Recommendations
Stage How It Works Scale
Candidate Generation A deep neural network takes the user’s watch history, search history, and demographics as input. It frames the problem as a multi-class classification: “Given this user’s history, which video will they watch next?” Uses collaborative filtering — users who watched similar videos get similar suggestions. 800M videos → ~500 candidates
Ranking Network A separate DNN scores each candidate using rich features: video age, channel authority, thumbnail CTR, previous watch %, user’s language, and time since last watch. Optimizes for expected watch time per impression — not learning quality. ~500 → top 20 shown

The Key Signals YouTube Uses

Signal What It Measures Weight
🕑 Watch Time Total seconds spent watching. Longer = better. The primary optimization target. Very High
👆 Click-Through Rate % of impressions that result in a click (driven by thumbnails & titles). High
📈 Audience Retention What % of the video was watched. 80%+ retention signals quality. Medium
👥 Collaborative Filtering “Users who watched X also watched Y.” Finds patterns across millions of users. Medium
👍 Engagement Likes, comments, shares, and subscribes after watching. Low–Medium
🔍 Search History Recent searches influence what appears in recommendations. Low

Why This Fails for Studying

When you’re watching a “Graph Algorithms” tutorial on YouTube, here’s what the algorithm actually optimizes for — and why it’s wrong for learning:

Problem What Happens to a Student
🐇 The Rabbit Hole You search for “Binary Search Trees.” YouTube recommends “10 Algorithm Tricks” → “Why I Left Google” → a tech podcast. Within 3 clicks, you’ve left learning entirely. The algorithm doesn’t distinguish learning intent from entertainment.
📏 No Difficulty Awareness YouTube has zero concept of difficulty levels. It recommends a PhD-level topology lecture after a “What is a variable?” tutorial — both have high watch time. No ELO, no ZPD, no progression.
🔄 Popularity Bias A viral “learn Python in 12 min” video (50M views) drowns out a rigorous 2-hour course (5K views). Collaborative filtering amplifies what’s popular, not what’s pedagogically sound.
🫧 Filter Bubbles Watch 5 JavaScript videos → YouTube keeps showing JavaScript. Even if your curriculum needs databases next. The algorithm reinforces your past, not your learning path.
📊 No Comprehension Signal YouTube doesn’t know if you understood the video — only that you watched it. A struggling student watching 3x looks identical to someone rewatching because they loved it.
🎯 No Learning Style Some learn best from live coding, others from visual diagrams. YouTube has no model for individual learning preferences — it shows what gets clicks from the general population.

🏗 Architecture

The Orion Profiling Engine

Multi-Dimensional Profile Composite (MDPC) – how Orion builds a rich, multi-faceted portrait of every learner.

The Core Idea

Instead of reducing a learner to a single score, MDPC builds a structured profile object with four independent dimensions. Each dimension uses the algorithm that's best suited for its specific job:

Analogy: Think of Spotify Wrapped. It doesn't just know you listened to 500 songs. It knows your genre preferences, your mood patterns, your peak listening hours, and which artists you discovered. MDPC does the same for learning.

Input: What Data We Collect

Every time a user watches a video and takes a quiz on Orion, we store the interaction in our Qdrant vector database with these fields:

Field Type Source Example
educational_level Ordinal (1—4) LLM Classification "Advanced"
practical_utility Categorical LLM Classification "Industry Application"
cognitive_load Ordinal (L/M/H) LLM Classification "High"
instructional_strategy Categorical LLM Classification "Live Coding"
content_density Ordinal (V/B/D) LLM Classification "Dense"
quiz_score Numeric (0—100) User action 85
subject_category Categorical LLM Classification "Computer Science"
vector 1024-dim embedding Computed [0.02, -0.11, …]
timestamp Unix epoch System 1707600000

Each video is classified using our Recommendation Agent – an LLM that extracts a structured taxonomy across 4 layers: Identity (what subject), Context (what level), Competency (what's taught), and Pedagogy (how it's taught).

Output: The 4 Dimensions

1
Semantic Centroid (EDCV)

A 1024-dimensional vector representing where in knowledge space the user currently lives. Algorithm: Exponentially-Decayed Centroid Vector.

2
Ability Rating (ELO)

A single float (800—2200) representing overall competence level. Algorithm: Modified ELO Rating System (from Chess).

3
Preference Distributions

Time-decayed frequency distributions answering "What kind of content does this user gravitate toward?". Algorithm: Decayed Frequency Analysis.

4
Comfort Zone

Score × context correlation identifying where the learner performs optimally. Algorithm: Bucket-Averaged Performance Mapping.

🧭 Dimension 1

Semantic Centroid (EDCV)

Exponentially-Decayed Centroid Vector – tracking where in knowledge space the user lives.

The Core Concept

Every video in Orion is represented as a 1024-dimensional embedding vector – a point in "meaning space" stored in our Qdrant vector database. Videos about similar topics cluster together; unrelated topics are far apart.

The EDCV computes a single point that represents the "center of gravity" of the user's recent learning – biased toward their most recent interactions.

🔗 Practical Application: Candidate Retrieval

How it works: We don't scan the entire database every time. We use the EDCV to query the Qdrant index for the nearest neighbors.

This step is purely about Recalling Candidates. It casts a wide net to fetch ~50 videos that are topically relevant (e.g., "Advanced Python Decorators") regardless of whether they are too hard, too easy, or mismatched in style.

Analogy: Imagine a satellite map of all human knowledge. The EDCV is like a GPS pin that drifts toward where you've been spending time lately. Watch 5 CS videos and it moves deep into CS territory. Start watching Physics and it gradually drifts. Stop watching CS for a month and the pin slowly releases its pull.

The Formula

Centroid Computation
centroid = Σ(weight_i Ã— vector_i) / Σ(weight_i)

Where each interaction's weight is computed as:

Weight Per Interaction
weight = score_factor Ã— e(-Î”t / half_life)

Breaking Down Each Component

1. Time Decay: e^(-Î”t / half_life)

This is an exponential decay function. The half-life is set to 30 days (2,592,000 seconds). Here's what this means in practice:

Time Since Interaction Decay Factor Interpretation
Just now (0 days) 1.000 Full influence
1 week ago 0.794 ~80% influence
30 days ago (1 half-life) 0.368 ~37% influence
60 days ago (2 half-lives) 0.135 ~14% influence
90 days ago 0.050 ~5% influence – nearly gone

2. Score Factor: Performance-based weighting

The quiz score determines how strongly a video "pulls" the centroid toward itself:

Quiz Score Score Factor Rationale
≥ 80% (Excelled) 1.2× Strong signal of genuine engagement and mastery
50—79% (Adequate) 1.0× Baseline – normal pull
< 50% (Struggled) 0.7× Topic didn't "stick" – weaker pull
-1 (Ungraded) 0.5× Watched but didn't attempt quiz – weakest signal

The Algorithm (Step by Step)

# From profile_engine.py – _compute_semantic_centroid()

centroid = [0.0] × 1024     # Initialize zero vector
total_weight = 0.0

for each interaction in user_history:
    vector = interaction.vector         # 1024-dim embedding
    Δt = now - interaction.timestamp    # seconds since interaction

    # Step 1: Compute time decay
    time_decay = e^(-Δt / 2,592,000)   # 30-day half-life

    # Step 2: Compute score weight
    score_factor = 1.2 if score ≥ 80
                   1.0 if score ≥ 50
                   0.7 if score < 50
                   0.5 if ungraded (-1)

    # Step 3: Compute combined weight
    weight = score_factor × time_decay

    # Step 4: Accumulate weighted vector
    centroid += weight × vector
    total_weight += weight

# Step 5: Normalize
centroid = centroid / total_weight
🔗 How Orion Uses This Dimension

Orion computes the Cosine Similarity between the User Centroid and every video in the database. This generates the initial candidate pool of ~50 relevant videos before any other filtering happens.

📐 Dimension 2

Ability Rating (ELO)

Dynamic Difficulty Adjustment – keeping learners in the Zone of Proximal Development.

The Core Concept

Just like in competitive chess or ranked video games, every user and every video has a numeric rating (starting at 1500). If a user answers a question correctly, their rating goes up and the video's goes down. If they struggle, their rating drops to find easier content.

🔗 Practical Application: Difficulty Filtering

How it works: Once we have the ~50 topically relevant candidates (from EDCV), we apply a hard filter based on ELO ratings.

We define a ±200 point range around the user's current rating. Any video outside this range is discarded. This ensures we never simply "match the topic" but fail on difficulty. It is the primary mechanism for ensuring the Zone of Proximal Development.

Analogy: A tennis ranking system. If you beat a Grand Slam champion, your rank skyrockets. If you lose to a beginner, your rank plummets. We use this to ensure users never see content that is too easy (boring) or too hard (frustrating).

The Formula

Expected Performance (Logistic Function)
Expected = 1 / (1 + 10(video_rating − user_rating) / 400)

Step-by-Step: What "Expected Score" means

1
Measure the Gap

We subtract your rating from the video's rating.
Gap = Video (1600) - User (1400) = +200

2
Calculate Probability

The formula converts this +200 gap into a percentage chance of success.
Expected Score = 0.24 (24%)

3
Set the Baseline

This 0.24 becomes the baseline. If you score higher than 24%, you gain points. If you score lower, you lose points.

💡
Why this formula? This is the standard Chess ELO logistic curve. The constant 400 means a 400-point difference predicts a 10x difference in win probability (91% vs 9%).

Note: 400 is a tunable hyperparameter. Lower values would make ratings swing faster (higher volatility), while higher values would make them stickier. For now, we use the Chess standard as a proven baseline.

Update Rule

Rating Update
user_rating += K × (actual − expected)

Where K (K-Factor) determines volatility:

  • K=32: New users (high volatility, fast placement).
  • K=16: Established users (stable rating).

Worked Example

📐 Hypothetical Edge Case: The Newbie vs. The Veteran

Scenario: Two users both watch the same "Advanced + High Load" video (Rated 1800) and score 85%.
How does the system react differently based on their history?

User A: The Newbie

Profile: Rating 1500 (Default), 0 graded interactions.
K-Factor: 32 (High volatility) 

Expected = 1 / (1 + 10^0.75) = 0.15
Actual   = 0.85

Δ = 32 × (0.85 - 0.15)
  = 32 × 0.70
  = +22.4 points

New Rating: 1522.4

Insight: System knows nothing about A, so it moves them fast to find their true level.

User B: The Veteran

Profile: Rating 1500, 50 graded interactions.
K-Factor: 16 (Stable) 

Expected = 1 / (1 + 10^0.75) = 0.15
Actual   = 0.85

Δ = 16 × (0.85 - 0.15)
  = 16 × 0.70
  = +11.2 points

New Rating: 1511.2

Insight: System "trusts" B's rating more. One good quiz doesn't disrupt their established history.

🔗 How Orion Uses This Dimension

Orion only recommends content within ±200 rating points of the user's ability. This keeps the learner in the Zone of Proximal Development – the educational sweet spot where content is challenging enough to learn from, but not so hard it's frustrating.

A user rated 1500 will see videos rated between 1300—1700 (Intermediate to Advanced range).

🎨 Dimension 3

Preference Distributions

Time-Decayed Frequency Analysis – understanding what kind of content each learner gravitates toward.

The Core Concept

Two learners can have the exact same ability rating yet need completely different content. One wants Live Coding tutorials for interview prep; the other wants Visual Explanations for academic research. ELO can't distinguish them – that's the job of Preference Distributions.

🔗 Practical Application: Ranking & Tie-Breaking

How it works: After ELO filtering, we might still have 20 "good" candidates. To decide the final order, we boost the scores of videos that align with the user's implicit preferences.

If a user shows a bias for "Live Coding" (Instructional Strategy), videos tagged as such get a +15% score boost during the re-ranking phase. This softly nudges the system toward the user's preferred teaching modality without strictly hiding other valid options.

Analogy: Think of a food delivery app. Knowing your "food rating" (4 stars) isn't enough. It also needs to know you prefer Italian over Thai, casual over fine dining, and lunch over dinner. Same idea: we track what kind of learning the user prefers across multiple axes.

The Three Axes We Track

Axis Possible Values Question Answered
Purpose
practical_utility		 Exam Prep, Industry Application, Academic Theory, Skill Building, General Interest Why are they learning?
Teaching Style
instructional_strategy		 Live Coding, Visual Explanation, Lecture, Scaffolded Tutorial, Problem-Based, Discussion How do they prefer to learn?
Subject
subject_category		 Computer Science, Mathematics, Physics, Biology, … What do they study?

The Formula

For each axis, we compute a time-decayed frequency distribution that sums to 1.0:

Decayed Count Per Category
count[value] += e(-Δt / half_life)
Normalization
distribution[value] = count[value] / Σ(all counts)

The preference half-life is 60 days (twice the centroid half-life) – preferences change more slowly than semantic interests.

The Algorithm

# From profile_engine.py – _compute_preferences()

for each field in [practical_utility, instructional_strategy, subject_category]:
    counts = {}

    for each interaction in user_history:
        value = interaction[field]    # e.g., "Live Coding"
        Δt   = now - interaction.timestamp
        decay = e^(-Δt / 5,184,000)   # 60 days in seconds
        counts[value] += decay

    # Normalize to sum = 1.0
    total = sum(counts.values())
    distribution = {k: v / total for k, v in counts}

Worked Example

📐 Hypothetical Edge Case: The Major Pivot

Scenario: User was a Biology Major (History: 50 Bio videos). Suddenly stops. Starts studying Computer Science (watched 5 CS videos this week).
Does the system cling to history or adapt?

Before Update (Historical Weight)

  • Biology Count: 50.0
  • CS Count: 0.0
  • Recommendation: 100% Biology

After 90 Days Gap + 5 CS Videos

# Step 1: Time Decay Old Interests
Bio Decay = 50 × e^(-90d / 60d)
          = 50 × 0.223
          = 11.15  (Faded significantly)

# Step 2: Add New Signal
CS Count  = 5.0 (Fresh, decay ≈ 1.0)

# Step 3: Normalize Distribution

🔗 How Orion Uses This Dimension

When multiple candidate videos have similar ELO ratings, Orion uses preferences as a tiebreaker. A "Live Coding" video about a new topic will rank higher for Bob than a "Lecture" on the same topic – because it matches his preferred teaching style.

Preferences also tune the Content Agent to align with the user's style: "This user prefers Live Coding for Skill Building. Prioritize practical, hands-on content."

⚡ Dimension 4

Comfort Zone

Score × Context Correlation – finding where each learner performs at their peak.

The Core Concept

Different learners perform differently under different conditions. Some thrive with dense, rapid-fire content. Others need slow, methodical explanations. The Comfort Zone identifies the environmental conditions under which each learner achieves their best results.

🔗 Practical Application: Delivery Tuning

How it works: This dimension doesn't just select content – it modifies how it is presented.

If a user has a "Low Cognitive Load" profile, the system instructs the Answering Chatbot to break answers into smaller bullet points and avoid jargon. It essentially customizes the UI/UX layer response generation to match the user's mental bandwidth at that moment.

Analogy: An athlete might sprint faster in cool weather and struggle in heat. We're finding the weather conditions under which this learner performs optimally – not just whether they're fast or slow overall.

Three Dimensions We Correlate

Dimension Buckets Question Answered
Cognitive Load Low, Medium, High How much mental effort can they handle?
Content Density Verbose, Balanced, Dense How compact should the material be?
Educational Level Foundational → Expert Which level yields the best scores?

The Algorithm

# From profile_engine.py – _compute_comfort_zone()

for each field in [cognitive_load, content_density, educational_level]:
    buckets = {}  # e.g., {"Low": [], "Medium": [], "High": []}

    for each graded interaction:   # skip ungraded (score = -1)
        bucket = interaction[field]
        buckets[bucket].append(quiz_score)

    # Compute average per bucket
    for each bucket_name, scores in buckets:
        avg = sum(scores) / len(scores)

    # Best = bucket with highest average
    best = argmax(buckets, by=average)

Worked Example

📐 Hypothetical Edge Case: The 'Fragile' Learner

Scenario: Carol is brilliant but gets overwhelmed easily.
History: High scores on Low/Medium load. One disaster on High load.

Cognitive Load Scores Average
Low 95, 98, 92 95.0%

🔗 How Orion Uses This Dimension

Orion synthesizes a deterministic profile context and dynamically tunes the context of the Answering Chatbot and Writing Agents.

For Carol: "This learner performs best with Medium cognitive load and Balanced content density. Break complex topics into manageable steps. Avoid overwhelming with too many concepts at once."

References & Next Steps

Deferred algorithms, roadmap, and academic foundations.

Algorithms Evaluated but Deferred

Bayesian Knowledge Tracing (BKT) – Phase 2

How it works: Models each concept (e.g., "Sorting") as Known/Unknown using 4 parameters: P(init), P(learn), P(guess), P(slip). Updates mastery probability via Bayes' rule after each interaction.

Why deferred: Binary state (no gradation from our 0—100 scores), requires pre-defined concept maps, assumes monotonic learning (never forgets), only tracks knowledge – not preferences.

Future value: "Alice has mastered Sorting (95%) but hasn't learned Graph Traversal yet (12%)"

Item Response Theory (IRT/MIRT) – Phase 3

How it works: Gold standard in psychometrics (SAT, GRE). Models P(correct) = f(ability, difficulty, discrimination). MIRT extends to multiple latent dimensions.

Why deferred: Needs 1000+ calibrated items. Our 0—100 scores need Graded Response Model adaptation.

Deep Knowledge Tracing (DKT) – Phase 4

How it works: LSTM/RNN neural networks process sequences of (skill, correctness) pairs. Hidden state encodes knowledge.

Why deferred: Needs 10,000+ interaction sequences. Black box – can't explain recommendations.
